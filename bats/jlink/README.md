# jlink.bat

## 📋 Overview

A specialized utility that creates a custom Java Runtime Environment (JRE) from a minified JRE, adding critical modules required for Java Agent support. This script was essential for enabling YPP-Gizmo to work after YPP's Java 8 → Java 25 migration required switching from the Java Accessibility Bridge to a Java Agent approach.

## 🎯 Problem Statement

Yahoo Puzzle Pirates uses a custom **minified JRE** that only includes modules necessary for the game. When the migration to Java 25 forced a switch to Java Agent-based instrumentation, a critical problem emerged:

- **Missing modules**: The minified JRE lacked `java.instrument` and other modules required for Java Agent loading
- **No agent support**: Without these modules, Java Agents couldn't attach to the running JVM
- **Unclear module list**: Determining which modules were in the minified JRE was non-trivial
- **Developer permission required**: Creating a custom JRE needed explicit approval from YPP developers

The challenge was to extract the module list from the minified JRE, then reconstruct it with the missing `java.instrument` module added.

## 💡 Solution

This script automates the custom JRE generation process:

1. **Module Discovery** - Reads the list of modules from `game-jre-modules.txt` (generated by a separate script)
2. **Module Enhancement** - Adds `java.instrument` to the module list
3. **JRE Compilation** - Uses `jlink` to rebuild the JRE with all required modules
4. **Optimization** - Applies compression and debug stripping to keep the JRE size reasonable
5. **Validation** - Confirms successful creation and provides next steps

## 🔧 How It Works

### Input: game-jre-modules.txt

This file contains all modules from the minified JRE:

```
java.base@25.0.1
java.datatransfer@25.0.1
java.desktop@25.0.1
java.logging@25.0.1
java.prefs@25.0.1
java.xml@25.0.1
jdk.accessibility@25.0.1
jdk.crypto.ec@25.0.1
jdk.unsupported@25.0.1
```

### Module Parsing

```batch
:: Build module list from file
set "MODULES="
for /f "tokens=1 delims=@" %%m in (game-jre-modules.txt) do (
    if not "%%m"=="" (
        if "!MODULES!"=="" (
            set "MODULES=%%m"
        ) else (
            set "MODULES=!MODULES!,%%m"
        )
    )
)
```

This extracts just the module names (before the `@` symbol) and creates a comma-separated list:
```
java.base,java.datatransfer,java.desktop,java.logging,java.prefs,java.xml,jdk.accessibility,jdk.crypto.ec,jdk.unsupported
```

### Adding java.instrument

```batch
set "MODULES=!MODULES!,java.instrument"
```

The critical `java.instrument` module is appended to enable dynamic Java Agent loading.

### JRE Compilation

```batch
"%JLINK%" --module-path "%SOURCE_JDK%\jmods" --add-modules !MODULES! --output "%OUTPUT%" --compress=2 --strip-debug
```

**Key options:**
- `--module-path` - Points to the full JDK's module directory (jmods)
- `--add-modules` - Specifies all modules to include in the custom JRE
- `--output` - Where to create the resulting custom JRE
- `--compress=2` - Applies ZIP compression to reduce size
- `--strip-debug` - Removes debug symbols to further reduce size

### Output

```
============================================
SUCCESS!

Custom JRE created at: C:\path\to\custom-runtime

This JRE includes all game modules PLUS java.instrument

Next steps:
1. Test it by replacing game's runtime folder
2. Or send to devs to integrate
============================================
```

## 📊 Technical Details

### Why java.instrument?

The `java.instrument` module provides:
- **JVM Instrumentation API** - Required for Java Agents to modify bytecode
- **Agent Loading** - Enables the `JVMTI.agent_load` command used in `attach.bat`
- **Dynamic Attachment** - Allows runtime attachment without restart

Without this module, Java Agents cannot function.

### Module Relationships

The custom JRE includes dependencies:
- `java.base` - Foundation module (always required)
- `java.desktop` - GUI support (for YPP client)
- `jdk.accessibility` - Accessibility features
- `java.logging` - Logging framework
- `java.xml` - XML processing
- Others - Various utilities the game depends on

When you add `java.instrument`, jlink automatically includes its dependencies too.

### Size Optimization

```batch
--compress=2 --strip-debug
```

- **Compression**: Reduces JRE size by ~30-40%
- **Debug stripping**: Removes symbol tables and debug info, further reducing size
- **Trade-off**: Smaller JRE, but less detailed stack traces

## 🛠️ Prerequisites

### Required Files
- **Full OpenJDK 25.0.1** - Must be a complete JDK (not just JRE) to access jlink and jmods
- **game-jre-modules.txt** - Generated by a separate module discovery script
- **jlink.bat** - This script

### System Requirements
- Windows system with Command Prompt
- ~1-2 GB free disk space (for temporary files during compilation)
- Administrative privileges may be needed for jlink

### Configuration

Update these paths in the script:

```batch
set "SOURCE_JDK=C:\Users\Admin\.jdks\openjdk-25.0.1"
set "OUTPUT=%~dp0custom-runtime"
```

- `SOURCE_JDK` - Path to your full OpenJDK installation
- `OUTPUT` - Where the custom JRE will be created

## 📌 Usage

```batch
jlink.bat
```

### Pre-execution Checklist

✓ `game-jre-modules.txt` exists in the same directory
✓ `SOURCE_JDK` path points to a real, full OpenJDK installation
✓ You have write permissions for the output directory
✓ At least 1-2 GB free disk space

### What It Does

1. **Validates** that jlink is available and game-jre-modules.txt exists
2. **Parses** the module list from the text file
3. **Adds** java.instrument to the modules
4. **Cleans up** any previous custom-runtime directory
5. **Compiles** the custom JRE using jlink
6. **Reports** success or failure

### Common Issues & Solutions

**"ERROR: jlink not found"**
- Verify `SOURCE_JDK` points to a full JDK (not a JRE)
- Check that the path exists and jlink.exe is in the bin directory

**"ERROR: game-jre-modules.txt not found"**
- Run the module discovery script first (e.g., `list-modules.bat`)
- Ensure the text file is in the same directory as jlink.bat

**"jlink failed"**
- Check that all modules in game-jre-modules.txt are valid
- Ensure you have enough free disk space
- Try running Command Prompt as Administrator

## 🚀 Evolution & Legacy Status

This script was **critical** for the initial Java 25 migration:
- Solved the "missing java.instrument" blocker
- Enabled a quick path to supporting Java Agents
- Allowed third-party tools like YPP-Gizmo to resume functioning

Later developments:
- YPP developers integrated `java.instrument` into the official minified JRE
- This script became unnecessary once the game's bundled JRE included the required modules
- The approach demonstrates understanding of Java module system and custom JRE generation
